# 测评与档案存储设计

**目标**：支持用户**多次重新测评**并**保留历史记录**，同时支撑「数据很多」的题目与结果，并为后续接真实数据库留好接口。

---

## 一、要解决的问题

1. **用户会反复做测评**：每次提交都要保留一条记录，不能只覆盖「当前结果」。
2. **数据量可能很大**：题目/选项多；历史记录会随时间增多。
3. **当前与历史要分清**：推荐逻辑用「当前档案」；归元/趋势用「历史记录」。

---

## 二、存储分工（先本地，后可选迁库）

| 用途 | 存什么 | 放哪里 | 说明 |
|------|--------|--------|------|
| **题目与选项（只读）** | 体质题、身体题、疼痛部位等 | **代码/JSON 文件** | 不占用户存储；您提供 Excel 后我落成 `assessment-data.ts` 或独立 JSON，体积大也可拆多个文件。 |
| **当前体质档案** | 最近一次测评的结果（体质类型、疼痛、时间等） | **本地键 `benyuan_profile`** | 与 key-code-index 一致；首页/炼体/食养读这个做推荐。 |
| **测评历史** | 每次测评的完整记录（时间、答案、算分结果） | **本地键 `benyuan_assessment_history`** | 数组，按时间倒序；保留最近 N 条（如 50），防止本地爆掉。 |

这样：**题目不进「数据库」**，只进代码或静态 JSON；**用户数据**才进「存储」（当前 + 历史）。

---

## 三、本地键与结构（与 key-code-index 对齐并扩展）

- **`benyuan_profile`**（已有）  
  - 含义：**当前**体质档案，供推荐使用。  
  - 结构：`{ constitutionType, bodyAnswers?, painTags?, constitutionAnswers?, timestamp?, ... }`  
  - 每次测评提交后：先追加一条到历史，再**用本次结果覆盖** profile。

- **`benyuan_assessment_history`**（新增，仅本地）  
  - 含义：**历史**测评记录，供归元/趋势/复测对比。  
  - 结构：数组，每项一条记录，例如：
    ```ts
    {
      id: string,           // 客户端生成的唯一 id（如 uuid）
      timestamp: number,
      constitutionAnswers: (number|string)[],
      constitutionType: string,
      bodyAnswers?: (number|string)[],
      painTags?: string[],
      bodyResult?: string,  // 身体测试结果（若您有）
      logicVersion?: string // 可选：当时使用的算分逻辑版本，便于回溯
    }
    ```
  - 策略：每次提交 **append**；读取时按 `timestamp` 倒序；保留最近 N 条（如 50），超出可删最旧的或您指定策略。

---

## 四、「数据库」选型建议（按阶段）

| 阶段 | 题目/静态数据 | 用户测评数据（当前+历史） | 说明 |
|------|----------------|---------------------------|------|
| **当前 MVP** | 代码 + TS/JSON 文件 | **uni 本地存储**（两个键） | 无需真实数据库；题目多就多文件或按模块拆；历史 cap 50 条。 |
| **历史再大 / 要查询** | 不变 | **IndexedDB**（H5）或 uni 兼容的本地库 | 容量大、可按时间/类型查；仍可不接后端。 |
| **多端同步 / 备份** | 可放 CDN 或后端 | **Supabase（PostgreSQL）** | 与《本元养生技术架构定稿版》一致：assessment_events 表 + user_profile 表；本地可先保留为缓存。 |

建议：**先按 MVP 做**——题目进代码/JSON，当前档案 + 历史进两个本地键；等您把题目和算分逻辑给全、产品跑顺，再视需要加 IndexedDB 或 Supabase，届时只需把「读/写 profile 与 history」换成调同一套接口（先本地实现，后换远程）。

---

## 五、流程小结

1. **您提供题目** → 我放进 `assessment-data.ts`（或 JSON），在知己页渲染；**不写库**。
2. **用户提交测评** → 调用算分逻辑得到本次结果 → **写入 `benyuan_assessment_history`（append）** → **用本次结果覆盖 `benyuan_profile`**。
3. **推荐/展示** → 读 `benyuan_profile` 用当前结果；归元/趋势读 `benyuan_assessment_history` 用历史。

这样既保留记录，又支持反复测评，数据多时题目在代码侧、历史在本地可控条数；后续要上真实数据库，只需替换「写/读」的实现即可。
