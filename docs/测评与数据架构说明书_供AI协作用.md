# 本元养生 · 测评与数据架构说明书（供 AI 协作用）

**文档用途**：供其他 AI（如 Claude Code、GPT 等）阅读后，从**数据架构师**与**产品架构师**角度给出专业建议或补充设计。人类开发者与主 AI（Cursor）将据此迭代实现。

**目标产品**：世界级健康管理 App + 微信小程序，预期**数万全球用户**；数据需**合理保存与复现**，兼顾**实时性**与**存储**；模块与逻辑、**输入输出关系**需想清楚，**数据架构需全盘考虑**。

**当前痛点**：用户完成 67 题体质测评后，结果「马上就没有了」，也不知道在哪里可以找到之前的测评结果；即便在测试阶段也不方便。希望重要事项提前全盘思考，可简化与迭代，但架构要经得起扩展。

## 一、产品与流程背景

### 1.1 测评整体结构（已确定）

「认识自己」拆成**三个部分**，用**步骤条（Step 1 → 2 → 3）**顺序完成；后续支持**单块更新**（从「我的测评档案」进入某一块单独修改）。

| 步骤 | 名称 | 内容 |
|------|------|------|
| **Step 1** | 基本信息 | 筛选题 1（特殊情况·多选）、题 2（年龄·单选）、题 3（性别·单选）、职业（可选） |
| **Step 2** | 中医体质测评 | 67 题（单选，1–5 分），九种体质判定、原始分/转化分 |
| **Step 3** | 运动/身体测评 | 题 4–19（见下），含 4–12 总分与评价、特别提醒、炼体法触发（活身法/动身法/合身法） |

- **筛选题 1**：有任一「特殊情况」时，**可做体质 67 题**，但**身体测试不建议做**；若用户仍要点进身体测试，弹窗（温和文案）提示后可选「继续填写」或「暂不填写」。
- **筛选题 2**：年龄过小或过大时，**可做体质 67 题和身体测试**；进入身体测试时弹窗提示「线上练习建议家人陪同」，可选「继续填写」或「暂不填写」。

### 1.2 身体测试（Step 3）题目与逻辑（题 4–19）

- **题 4**：健康的重要性（单选，0/1/2 分）。
- **题 5**：每周运动次数（单选，0/1/2/3 分）。
- **题 6**：运动类型（**多选**，18 项；用户只看到「选项 + 强度」标签，每题有内部分数；得分 = 所选选项分数之和）。
- **题 7/8/9**：低/中/高强度每次运动时长（单选，各有分数）。
- **题 10**：运动后感觉（单选，1/3/5/4 分）。
- **题 11**：运动后次日身体状态（单选，5/4/2/1 分）。
- **题 12**：关节健康状况（单选，5/3/2/1 分）。
- **4–12 总分** = 题 4 得分 + … + 题 12 得分。  
  **评价**：≤14 → 运动表现较弱…；15–25 → 运动表现良好…；≥26 → 运动表现佳…。  
  **特别提醒**：若题 9 得分=2 或 题 10 得分=1 或 题 11 得分=1，展示「过度锻炼、透支…调养与锻炼结合」；该提醒在**结果页**展示，并在**炼体页或首页**做弱提示（可关闭；下次测评不满足则不再显示）。
- **题 13**：炼体法·长期困扰（多选，7 项，每题 1 分）；**先采集与展示，暂不参与推荐逻辑**。
- **题 14**：炼体法·中式运动核心需求（多选，4 项，每题 1 分）；同上，先采集与展示。
- **题 15/16**：身体活动度/舒展、久坐僵硬（四档：从未/轻微/中等/明显）；选「中等」或「明显」→ **本元炼体法触发，内容 = 活身法**。
- **题 17/18**：不会用力易伤、发力模式（四档同上）；选「中等」或「明显」→ **触发，内容 = 动身法**。
- **题 19**：身体深层开发不够（四档同上）；选「中等」或「明显」→ **触发，内容 = 合身法**。  
  若多题触发，结果页**并列展示**所有内容（如「为您推荐：活身法、动身法」）。

### 1.3 交互与体验（已确定）

- 全测评内**选择题样式与颜色统一**（同一套单选/多选组件与样式变量）。
- 单选 = 选一个顶掉上一个；多选 = 可勾选多项。业界通用表现即可。
- **单块更新**：提供「我的测评档案」页，三块并列展示当前摘要，每块有「更新」按钮；点更新只进入该块流程，提交后只更新该块并回到档案或结果，不重做其他块。

## 二、当前实现状态（代码与存储）

### 2.1 技术栈与存储

- **客户端**：uni-app（Vue 3），一套代码覆盖 **App + 微信小程序**。
- **本地存储**：`uni.setStorageSync` / `uni.getStorageSync`；键名见 `docs/key-code-index.md` 与下文。
- **当前**：无登录时数据仅本地；未来注册后计划用 Supabase 等做云端同步。

### 2.2 现有存储键与结构

- **`benyuan_profile`**（当前档案）：  
  现有字段包括 `constitutionAnswers`、`constitutionType`、`constitutionDimensionScores`、`constitutionDimensionConvertedScores`、`constitutionTendency`、`bodyAnswers`、`painTags`、`timestamp`。  
  **尚无**：筛选题 1/2、年龄、性别、职业、题 4–19 的完整结构、4–12 总分、炼体法触发内容、题 13/14 多选等。
- **`benyuan_assessment_history`**（测评历史）：  
  数组，每项为一次测评记录；现有结构侧重**体质**（id、timestamp、constitutionAnswers、constitutionType、baseScores、actualScores、各维度分、tendency 等），**尚无**分块类型（basic_info / constitution / body_test）、也无题 4–19 的 payload。
- **历史条数**：当前实现保留最近 50 条（`HISTORY_CAP = 50`），append 后 slice。

### 2.3 相关代码路径（app 目录为 uni-app 前端）

| 路径 | 职责 |
|------|------|
| `app/src/lib/profile-utils.ts` | 读写 `benyuan_profile`、`benyuan_assessment_history`；体质算分、转化分、判定；`saveProfile`、`appendAssessmentHistory`、`getProfile`、`getAssessmentHistory` |
| `app/src/lib/assessment-data.ts` | 体质 67 题题目与选项（CONSTITUTION_QUESTIONS）；BODY_QUESTIONS 等目前为空，待扩展题 4–19 |
| `app/src/pages/assessment/assessment.vue` | 测评页：当前为体质 67 题 + 体质结果页；身体测试为占位（原「5 题 + 疼痛部位」为 mock，不再使用） |

## 三、需要其他 AI 协助回答的问题

请从**数据架构**与**产品/信息架构**两方面给出可落地的建议（可简化分阶段，但重要决策需说明理由）。

### 3.1 数据架构

1. **当前档案（profile）的完整 schema 建议**  
   - 如何组织「基本信息」「体质」「身体测试」三块，便于：  
     - 分块更新（只写某一块）、  
     - 与历史记录一致（历史中每条的 payload 是否与 profile 子结构一一对应）、  
     - 未来同步到云端（如 Supabase 表结构或 JSON 列）？  
   - 是否需要每块独立的 `*UpdatedAt` 时间戳？是否需要 `lastFullAssessmentAt`（最后一次完整三步）？

2. **测评历史（assessment_history）的 schema 与语义**  
   - 每次「提交」应记录的类型：`basic_info` | `constitution` | `body_test` | `full` 是否足够？  
   - 单条记录的 `payload` 应包含哪些字段，才能**完整复现当次结果**（例如体质 67 题答案 + 各维度分 + 判定文案；身体测试题 4–19 答案 + 4–12 总分 + 评价 + 炼体法内容列表）？  
   - 历史条数上限（如 50）与本地容量、未来云端同步策略如何平衡？是否需要「分页」或「按类型筛选」？

3. **身体测试（题 4–19）的答案存储格式**  
   - 题 4–12：多为单选，少量多选（题 6）；题 13/14 多选；题 15–19 单选四档。  
   - 建议：在 profile 与 history.payload 中采用何种结构（例如 `q4`–`q19` 的扁平键，或按题组分组），便于读写、版本演进与云端同步？

4. **实时性与一致性**  
   - 读：首页、炼体页、食养、结果页等应读 profile 的哪些字段、何时读（每次 onShow？还是事件驱动）？  
   - 写：提交某一块时，是「先写 profile 再 append history」还是「先 append history 再根据 history 最新一条更新 profile」？若多端同步，冲突策略（last-write-wins per block？）建议？

### 3.2 产品/信息架构

5. **「结果在哪里」的入口与路径**  
   - 用户完成某一步（如 67 题）后：  
     - 当次结果应在哪个页面、以何种形式展示？  
     - 「当前」整体档案应在哪里查看（建议的页面名称与入口，如首页卡片、底部导航、设置内「我的档案」）？  
     - 「历史某次」结果应在哪里查看（列表 + 详情；列表按时间还是按类型筛选）？  
   - 请给出**具体页面/路由与数据来源**（读 profile vs 读 history[i].payload）的对应关系。

6. **「我的测评档案」页的信息结构**  
   - 三块（基本信息、体质、身体测试）各展示哪些摘要信息（一行或两行）？  
   - 每块「更新」按钮的跳转目标：是否进入同一测评页但限定为「仅该块」（如 step=1 或 mode=basic_info_only）？更新后返回档案页还是结果页？

7. **与现有模块的输入输出关系**  
   - 筛选题 1/2 的结果：除控制「是否弹窗建议不做身体测试/家人陪同」外，是否还需要写入 profile 供其他模块（如首页、炼体）使用？  
   - 体质结果：当前已被食养、知识等引用；新增「身体测试」后，炼体页推荐是否同时依赖「体质 + 炼体法内容列表 + 4–12 评价」？请用简短输入输出表或依赖图说明。

## 四、约束与可接受范围

- **MVP 先本地**：可先不实现云端同步，但 schema 与读写路径需为后续同步预留（例如 profile 与 history 结构可直接映射为云端表或文档）。
- **兼容现有体质逻辑**：已有 67 题算分、转化分、平和质/偏颇质判定、`CONSTITUTION_ENTRY_COUNTS` 等保持不变；新增的是「三块结构、基本信息、题 4–19、历史分块类型与 payload」。
- **文档与代码一致**：采纳其他 AI 的建议后，将更新本文档及 `app/docs/` 下相关说明（如测评与档案存储设计），并在 `app/src/lib/profile-utils.ts`、`assessment-data.ts`、`assessment.vue` 中实现。

## 五、附录：关键业务规则速查

- **题 1**：多选；若选了任一项「特殊情况」（非「没有上述情况」）→ 体质可做，身体测试不建议做，进入身体测试时弹窗（温和文案）。
- **题 2**：单选年龄区间；&lt;12 或 &gt;70 → 体质和身体测试都可做，进入身体测试时弹窗「家人陪同」。
- **题 6**：18 项多选；用户见「选项 + 强度」；内部每题有分数；本题得分 = 所选选项分数之和。
- **题 12**：合计 = 本题四选项得分之和（对应 Excel G117:G120）。
- **4–12 总分**：题 4 至题 12 每题得分相加；评价分段 ≤14 / 15–25 / ≥26；特别提醒条件：题 9 得分=2 或 题 10=1 或 题 11=1。
- **炼体法 15–19**：每题四档；选「中等」或「明显」即触发；15/16→活身法，17/18→动身法，19→合身法；可多题触发，并列展示。
