## 二、已确认规则（供实现时遵守）

### 食养·食谱第一层分类与展示、推荐选品

- **第一层分类**：食谱分为 **早餐** 与 **非早餐**；非早餐包括 **食疗方** 与 **家常菜**（CSV 列「类别（食疗方/家常菜/早餐）」）。
- **体质与选品**：**任何体质都可以吃全部早餐食谱和全部家常菜食谱**（不做体质筛选）。仅 **食疗方** 按体质规则：**测评结果包含平和质**（是/基本是平和质）→ 全部食疗方可用；**测评结果不包含平和质** → 用户体质集合 = type + tendency，仅当食谱「适合体质」**完全包含**该集合时展示/推荐（用户体质集合 ⊆ 食谱适合体质）。
- **完整食谱展示顺序**：1）早餐；2）食疗方（项下：菜、主食、汤）；3）家常菜（项下：菜、主食、汤）。
- **周食谱与每日食谱推荐选品**：
  - **早餐**餐位：全部早餐可选（不限体质）。
  - **午餐、晚餐**餐位：全部家常菜可选；食疗方按上条规则（平和质全部，偏颇按「用户体质集合 ⊆ 食谱适合体质」筛选）。
- 实现：`recipe-store` 中 `getRecipesByConstitutionDishTypeAndSlot(..., shiliaoUserSet)` 早餐位返回全部早餐，午/晚餐返回全部家常菜 + 按 `shiliaoUserSet` 筛选的食疗方（`shiliaoUserSet` 来自 `getShiliaoUserSet(profile)`，为 `null` 时食疗方全选）；食谱库中早餐用 `getAllBreakfastRecipes()`，家常菜用 `getAllJiachangRecipes()`，食疗方用 `getShiliaoRecipesForProfile(getShiliaoUserSet(profile))`。

### 食养·每餐结构

- 主食：最多 1 道  
- 汤：最多 1 道  
- 菜：至少 1 道  

### 食养·重复与个性化

- 按「日期 + 用户」维度生成/推荐。  
- 建议：同一餐位（如早餐）在 **3 天内不重复** 同一道菜/食谱。

### 食养·日菜谱冲突校验

- **仅当日、不跨天**：冲突校验只针对**当天**的早+午+晚三餐内检查，不比较不同日期之间的菜谱。
- **无论是用户自选还是系统推荐**，在生成或确认「日菜谱」（单日或当餐组合）时，都必须做冲突检查：
  - **不宜同吃**：同一餐或同一天内的多道菜之间，不能出现「甲食谱的食材」出现在「乙食谱的不宜同吃」里（或反向）。数据来源：CSV 列「不宜同吃」+ 各道菜的「食材」。
  - **烹饪方法**（若后续有该维度）：同一餐内若有不宜搭配的烹饪方式，也需避免或提示。
- 若检测到冲突：应提示用户并建议替换其中一道，或由推荐逻辑直接排除冲突组合。

### 食养·选择/推荐时机与周计划

- **周计划（推荐节奏）**  
  - 每周六上午：用户自选或接受推荐「下一周」的菜谱（下周一～下周日），并推荐对应食材，方便周末采购。  
  - 可设置提醒（如周六上午）。  
  - 选择/推荐界面按「下周一、下周二……下周日」逐日展示。

- **有周计划时**  
  - 每天：在前一天下午或晚饭时间推送当日菜谱（提醒/确认）。

- **未做周计划时**  
  - 前一天下午或晚饭时间推送「第二天」的菜谱推荐，规则同上（每餐结构 + 3 天不重复）。
  - **确认**：若用户不规划下周菜单（未确认周计划或选择「不安排下周菜单」），则当日、次日无周计划可用，**今日菜单与明日菜单均为「每天滚动推荐」**（今日/明日缓存或即时单日推荐）。实现上：`getTodayMealsFromPlan()` / `getDayMealsFromPlan(tomorrow)` 无数据时，即走缓存与单日推荐。

### 食养·菜谱展示字段（按场景）

CSV 中除名字外的主要字段：**食材、烹饪步骤、归经、类别（食疗方/家常菜/早餐）、功效、适合病证、适合体质、不宜同吃、食用方法、注意事项**，以及多种体质列。

建议按场景展示如下（实现时可按文案长度做截断或「展开」）：

| 场景 | 建议展示字段 | 说明 |
|------|--------------|------|
| **列表/卡片**（选择、推荐、推送列表） | 名字、**类别**、**适合体质**（简短）、**功效**（一句或前 N 字） | 便于快速判断「是不是我要的」「是否合体质」；功效可摘要。 |
| **详情/做菜时**（点进某道菜） | 名字、**食材**、**烹饪步骤**、功效、适合体质、**不宜同吃**、注意事项、食用方法（若有） | 做菜必看：食材+步骤；安全与注意：不宜同吃、注意事项。 |
| **周计划 & 采购** | 名字、**食材**（可汇总一周去重） | 采购清单以食材为主；名字用于对应到某天某餐。 |

- **可不直接对用户展示**（仅用于推荐或后台）：归经、适合病证、体制长度、各体质 0/1 列等，用于筛选与匹配即可。
- 若「适合体质」在列表里过长，可只显示「适合 X 质」或前几条；详情页再展示完整。

- **「你的饮食概览」展示结构（已调整，不再使用 Lovable/DIET_GUIDE）**  
  - **1. 体质测试结果**：来自档案 `profile.constitution.summary` 或 `profile.constitution.type`（未测评时用 mock 体质），直接展示，不做「适合食材/少吃」等衍生文案。**来源**：测评完成后 `getConstitutionDetermination(convertedScores)` 得到 `summary`（如「基本是平和体质，有阴虚、气虚倾向」）与 `primary`，存档于 `profile.constitution`；食养页读取后展示。
- **「适合你的食谱库」共 N 道**：N = 早餐总数 + 家常菜总数 + 可用食疗方数。早餐 = `getAllBreakfastRecipes().length`，家常菜 = `getAllJiachangRecipes().length`，食疗方 = `getShiliaoRecipesForProfile(getShiliaoUserSet(profile)).length`（平和质用户为全部食疗方，偏颇用户为「适合体质」完全包含其体质集合的食疗方）。  
  - **2. 五运六气 · 节气**：来自 `app/src/lib/solar-terms.ts`（`getCurrentSolarTerm`、`getWuYunLiuQi`）。节气当前为公历近似（固定月日），约有 1～2 日误差；五运六气为占位「待接入」。数据源方案（预计算 5～10 年 vs API）见待办「食养·节气与五运六气数据源」。  
  - **3. 食养之道**：预留区域，内容将根据 **体质 + 五运六气 + 节气** 由合伙人提供 CSV 后生成；当前占位文案：「内容将根据体质、五运六气与节气由合伙人提供 CSV 后生成」。接口 `getShiyangAdvice(constitutionType, solarTermName, wuYunLiuQi)` 在 `diet-data.ts`，待 CSV 接入后查表返回。  
  - **4. 不再展示**：不在此处展示「适合的食材」「少吃」；若需要，可纳入上述第 3 点食养之道文案中，由 CSV 统一提供。

### 食养·MLLM 使用策略（产品/技术对齐用）

- **当前不采用 MLLM**  
  - 食养核心流程（推荐菜谱、周计划、冲突校验、展示字段）依赖规则 + CSV 数据即可完成，不依赖图像或多模态。  
  - 不在推荐/冲突校验等核心逻辑中接入 MLLM，避免成本、延迟与识别误差影响主流程。

- **未来可考虑的 MLLM 场景**（延后评估，非当前范围）  
  | 场景 | 可能用途 | 注意点 |
  |------|----------|--------|
  | 拍照记一餐 | 用户拍餐盘 → 识别菜品/食材 → 记入「今日吃了什么」或匹配到食谱 | 识别有误属正常，需标注「仅供参考」；涉及图片上传与隐私。 |

  - 若引入：需单独评估成本、准确度、隐私与合规，且上述能力作为**可选增强**，不替代现有推荐与冲突校验逻辑。

## 四、食养·食谱主数据来源与使用（源头对齐）

- **唯一源头**：**`app/docs/recipe-master.csv`** 为食谱的**主数据（master）**，所有「适合体质」推荐、食谱库、不宜同吃校验均只使用该表的数据；不在此 CSV 中的食谱不会出现在应用内。
- **使用链路**：  
  1. **维护**：由产品/合伙人维护 `recipe-master.csv`（增删改食谱、适合体质、不宜同吃等）。  
  2. **生成**：在项目根目录执行 `node app/scripts/csv-to-recipe-json.cjs`，读取上述 CSV，生成 **`app/src/lib/recipe-master.json`**。  
  3. **运行态**：前端 **`app/src/lib/recipe-store.ts`** 仅加载 **`recipe-master.json`**（不直接读 CSV），据此做按体质筛选、按类型推断（主食/汤/菜）、不宜同吃校验等；所有推荐均为该全集的子集。
- **CSV 列 → 代码字段**（脚本 `csv-to-recipe-json.cjs` 的映射，列名须与 CSV 表头一致）：  

  | CSV 列名 | 代码字段 | 说明 |
  |----------|----------|------|

## 五、食养·其他数据结构与存储（供多用户/扩展参考）

- **食谱全集**：见第四节「食谱主数据来源与使用」。
- **周计划**：本地 key `benyuan_diet_week_plan`。结构：`{ schemaVersion, weekStartYMD, meals: { [dateYMD]: { breakfast, lunch, dinner } }, updatedAt, userId? }`。当前单用户；多用户时可在 key 加 userId 或存为 `{ [userId]: WeekPlanRecord }`。
- **单日推荐缓存**：key `benyuan_diet_today_generated`，存当日生成的单日菜谱（无周计划时用），避免每次进入重新随机。
- **已完成/餐后反馈**：沿用 `benyuan_diet_completed`、`benyuan_diet_feelings`（见 diet-utils），按餐位（breakfast/lunch/dinner）与日期记录。

## 六、食养·产品对齐（用户问答与进展）

### 1. 换一道 + 周计划逻辑（已对齐）

- **有周计划时**：打开「每天」的早/中/晚饭，看到的 = 已确认的周计划里**这一天**的食谱；若用户点「换一道」并选新菜，会**同步更新周计划里当天的数据并保存**。
- **无周计划时**：今日三餐来自「单日推荐」缓存；换一道只更新该缓存。

### 2. 「共 N 道」计算逻辑（已实现并文档化）

- **来源**：页面上的「共 N 道」= 全部早餐 + 全部家常菜 + 可用食疗方。即 `fullRecipeList.length`，其中 `fullRecipeList` = `getAllBreakfastRecipes()` + `getAllJiachangRecipes()` + `getShiliaoRecipesForProfile(getShiliaoUserSet(getProfile()))`。
- **步骤**：① 早餐与家常菜不做体质筛选，全部计入；② 食疗方：若用户为平和质（`getShiliaoUserSet` 返回 `null`）则全部食疗方计入；若为偏颇体质则仅「适合体质」**完全包含**用户体质集合（type + tendency）的食疗方计入。详见 `docs/食养-食疗方筛选逻辑与示例.md`。

### 3. 食谱库分类展示（已实现）

- **入口**：「查看全部食谱」弹层内。
- **结构**：① **早餐**（一整块）；② **家常菜** → 下分「菜」「主食」「汤」；③ **食疗方** → 下分「菜」「主食」「汤」。
- **实现**：使用 CSV 列「类别（食疗方/家常菜/早餐）」+ 名字关键词推断的 `inferDishType`（菜/主食/汤），见 `diet.vue` 中 `recipeLibraryByCategory`。

### 4. 「每条餐卡」含义

- 指**今日三餐**区块里，每一块代表一餐的卡片：**早餐**一张卡、**午餐**一张卡、**晚餐**一张卡。  
- 若要做「推荐 / 自选」标签：即在每张餐卡上标出该餐是系统推荐还是用户换过菜（自选），需在数据层区分「该餐是否被用户替换过」，当前为可选优化。

### 5. 周计划存档（已实现：周日 24:00 归档，最多 26 周）

- **触发**：进入食养页时调用 `ensureArchiveCurrentPlan()`。若当前存储的周计划其「该周周日」已过（即今日日期 > 该周周日 YYYY-MM-DD，视为周日 24:00 已过），则将该周计划写入归档并清空当前计划。
- **归档存储**：key `benyuan_diet_week_archive`，结构 `{ [weekStartYMD]: WeekPlanRecord }`。每次归档后删除 `weekStartYMD` 早于「当前日期 − 26 周」的记录，只保留最近 26 周。

## 七、食养·「共 N 道」计算逻辑（细粒度验算用）

页面「适合你的食谱库」中「共 **N** 道」的 **N** = 全部早餐 + 全部家常菜 + 可用食疗方数量。早餐与家常菜不做体质筛选；食疗方按「用户体质集合」规则筛选。

### 步骤 1：早餐与家常菜数量

- **代码**：`getAllBreakfastRecipes().length`、`getAllJiachangRecipes().length`
- **说明**：任何体质均使用全部早餐、全部家常菜，数量与体质无关。

### 步骤 2：食疗方「用户体质集合」

- **代码位置**：`app/src/lib/diet-mock.ts` → `getShiliaoUserSet(profile)`
- **输入**：`profile` = `getProfile()`
- **逻辑**：若 `profile.constitution.type === '平和质'` → 返回 `null`（全部食疗方可用）；否则返回 `[type, ...(tendency || [])]`，即偏颇体质 + 所有倾向体质。
- **输出**：`null` 或字符串数组，供食疗方筛选使用。

### 步骤 3：可用食疗方数量

- **代码位置**：`app/src/lib/recipe-store.ts` → `getShiliaoRecipesForProfile(shiliaoUserSet)`
- **逻辑**：若 `shiliaoUserSet` 为 `null` 或空 → 返回全部食疗方；否则仅返回满足「用户体质集合 ⊆ 食谱适合体质」的食疗方（`recipe-types.ts` 中 `recipeSuitsConstitutionSet(recipe, userSet)`）。
- **输出**：食疗方列表，其 length 为可用食疗方数量。

### 步骤 4：页面 N

- **代码位置**：`app/src/pages/diet/diet.vue`
- **数据**：`fullRecipeList = computed(() => [...getAllBreakfastRecipes(), ...getAllJiachangRecipes(), ...getShiliaoRecipesForProfile(getShiliaoUserSet(getProfile()))])`
- **展示**：`共 {{ fullRecipeList.length }} 道`，即 **N = fullRecipeList.length**。

### 验算核对建议

1. **早餐/家常菜**：与 CSV 中类别为「早餐」「家常菜」的行数一致（且脚本生成 JSON 有效）。
2. **平和质用户**：食疗方数量 = CSV 中类别为「食疗方」的行数。
3. **偏颇用户**：食疗方数量 = 满足「适合体质」列完全包含其 type + tendency 的食疗方行数；可逐条用 `recipeSuitsConstitutionSet(recipe, userSet)` 核对。
